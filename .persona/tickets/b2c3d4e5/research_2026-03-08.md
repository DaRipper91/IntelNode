# Research: PRoot Command Builder and Atomic Extraction

**Date**: 2026-03-08

## 1. Executive Summary
This research maps the current implementation of PRoot command construction and rootfs extraction. The current system relies on a massive, hardcoded static string for booting and a non-atomic extraction process that is vulnerable to corruption.

## 2. Technical Context
- **`lib/workflow.dart:739`**: `D.boot` defines the static base PRoot command.
- **`lib/workflow.dart:1165-1180`**: `launchCurrentContainer` assembles the final command by exporting environment variables and triggering `D.boot`.
- **`lib/workflow.dart:966-988`**: `initForFirstTime` extracts the rootfs directly into `containers/0`.
- **`lib/models.dart`**: Contains `ContainerInfo` and `CommandInfo` data classes.

## 3. Findings & Analysis
### PRoot Command Construction
The `boot` command is currently a single string:
`PROOT_NO_SECCOMP=1 \$DATA_DIR/bin/proot -H --change-id=1000:1000 ...`
It relies on shell-side variable expansion (`$DATA_DIR`, `$CONTAINER_DIR`, `$EXTRA_MOUNT`, `$EXTRA_OPT`) performed via `Util.termWrite`.

### Extraction Logic
In `initForFirstTime`, the extraction is handled by:
`\$DATA_DIR/bin/proot --link2symlink sh -c "cat xa* | \$DATA_DIR/bin/tar x -z --delay-directory-restore --preserve-permissions -v -C containers/0"`
If this process fails mid-way, the `containers/0` directory remains partially populated, preventing a clean retry without manual intervention or a full app data wipe.

### Integration Points
- **`lib/models.dart`**: Suitable for a `ProotCommandBuilder` class that manages flags, mounts, and environment variables.
- **`Workflow.initForFirstTime`**: Needs to be refactored to use a temporary directory (e.g., `containers/0_tmp`) and an atomic `mv` or `rename` operation.

## 4. Technical Constraints
- **Shell Escaping**: Command construction must handle Dart-to-Shell escaping correctly.
- **Permissions**: Extraction requires `proot` with `--link2symlink` to correctly handle permissions on Android's `No-Exec` storage partitions.

## 5. Architecture Documentation
- **Dynamic Assembly**: Final execution strings are built at runtime based on `GlobalSettings` (VirGL, HiDPI, Turnip).
- **UID Mapping**: Post-extraction shell logic maps Android/Termux UIDs to the container's `/etc/passwd`.
